# Going public, and shipping v0.1.0

The maintainer's runbook: every step from a private repo to a public release
that installs with one line, `brew`, `npm i -g tazamun`, or `cargo install
tazamun` — in the order that works. Steps marked **[done]** were completed in
the release-readiness commit this file arrived in; everything else is yours,
because it needs your accounts.

The state of things: the release pipeline is cargo-dist. A pushed tag that
looks like a version builds every platform, checksums and attests the
artifacts, creates the GitHub Release, regenerates the shell and PowerShell
installers, publishes the Homebrew formula to your tap, and publishes the npm
package. crates.io is the one manual publish. `tazamun update` then works
against the release assets — the archive layout and asset names are pinned by
unit tests in `src/cli.rs`.

---

## 0. What was already fixed [done]

Three things silently blocked any release; they are corrected in this commit:

- **Every past Release run queued for 24 hours and died.** The workflow was
  generated by dist 0.28 against `ubuntu-20.04`, a runner label GitHub has
  retired — the jobs could never start. The workflow is regenerated with dist
  0.32.0 on live labels (`ubuntu-22.04`, `windows-2022`, `macos-14`,
  `macos-15-intel`), which also means macOS binaries ship for both
  architectures now.
- **`tazamun update` could download but never install.** dist archives nest
  the binary under `tazamun-<target>/`; the updater looked for it at the
  archive root. Fixed with `bin_path_in_archive`, proven by extracting the
  real locally-built artifact through the exact configured path, and pinned
  by `cli::tests::update_bin_path_matches_the_dist_archive_layout`.
- **A hand-built Windows GNU binary could never find an update.** Releases
  ship MSVC; the updater now normalises `x86_64-pc-windows-gnu` to the MSVC
  asset, so side-loaded builds graduate to the released ABI on first update.

Also in place: the npm package is **unscoped `tazamun`** (the name was free on
npm as of 2026-07-20 — as was the crate name on crates.io), npm publishing is
wired into the release (`publish-jobs = ["homebrew", "npm"]`), and
`cargo package` has verified the crate compiles exactly as crates.io will
build it.

## 1. Accounts and tokens (before anything else)

Three externals, all needed before the first tag:

**npm.** Create/log into the npm account that will own `tazamun`, mint an
**Automation** token (Access Tokens → Generate → Automation), then:

```bash
gh secret set NPM_TOKEN --repo cc1a2b/tazamun
```

The release workflow's npm job publishes with it. No token → that one job
fails; everything else still releases.

**Homebrew tap.** The release pushes a formula to `cc1a2b/homebrew-tap`:

```bash
gh repo create cc1a2b/homebrew-tap --public \
  --description "Homebrew tap for tazamun"
```

Mint a fine-grained PAT scoped to only that repo with **Contents:
read-and-write**, then:

```bash
gh secret set HOMEBREW_TAP_TOKEN --repo cc1a2b/tazamun
```

**crates.io.** Log in with GitHub at crates.io, mint an API token, and keep it
for step 6 — this one is used from your machine, not from CI.

## 2. Fix the main CI before anyone can see it

`ci.yml` still targets your self-hosted runners, and both are offline — so
every push shows a forever-queued check. Worse than cosmetic: **on a public
repo, a pull request from a fork that reaches a self-hosted runner is
arbitrary code execution on your machine.** GitHub's own guidance is blunt:
do not use self-hosted runners on public repos.

Public repos get **unlimited free** GitHub-hosted minutes, which removes the
reason the self-hosted runners existed. The switch:

```bash
cd ~/projects/tazamun
sed -i 's/runs-on: \[self-hosted, linux, x64\]/runs-on: ubuntu-latest/' .github/workflows/ci.yml
sed -i 's/runs-on: \[self-hosted, "\${{ matrix.os }}", x64\]/runs-on: ${{ matrix.os }}/' .github/workflows/ci.yml
```

then edit the matrix values from `linux` / `windows` to `ubuntu-latest` /
`windows-latest`, and read the file once top to bottom for anything that
assumed a pre-provisioned machine (cached toolchains, local paths). If you
keep the self-hosted runners for private experiments, at minimum set
Settings → Actions → *Require approval for all external contributors*.

## 3. Repository settings

In Settings, before the flip:

- **Security → Private vulnerability reporting: enable.** SECURITY.md points
  people here; without it the link dead-ends.
- **Security → Secret scanning + push protection: enable** (free on public).
- **General → Description / topics:** the About box is the first thing search
  shows. Suggested topics: `rust`, `p2p`, `sync`, `file-sync`,
  `end-to-end-encryption`, `iroh`, `quic`, `cli`, `egui`.
- **Personal exposure, decide deliberately:** one authored name and one email
  address exist in the project — the disclosure email in
  `docs/THREAT_MODEL.md` §9, which going public turns into a spam magnet.
  Consider pointing §9 at GitHub private advisories (SECURITY.md already
  does). The git history is a single commit authored `cc1a2b`, so there is
  nothing to scrub there.

## 4. Flip it

```bash
gh repo edit cc1a2b/tazamun --visibility public --accept-visibility-change-consequences
```

(Or Settings → General → Danger Zone → Change visibility.) Flip **before**
pushing the release tag: hosted macOS and Windows minutes are billed on a
private repo and free on a public one.

Forks, stars and watchers reset on a visibility change; releases and issues
survive. There are no releases yet, so nothing stale becomes visible.

## 5. Ship v0.1.0

The tag must point at a commit whose workflows are the fixed ones — the tag
currently on the remote points at the old, unrunnable release.yml. Move it:

```bash
cd ~/projects/tazamun
git push origin main
git tag -fa v0.1.0 -m "Tazamun v0.1.0" HEAD
git push -f origin v0.1.0
```

The forced tag push triggers the Release workflow. Twenty minutes later the
release page should hold, per platform, `tazamun-<target>.tar.gz` (`.zip` on
Windows) plus `.sha256` files, `sha256.sum`, `tazamun-installer.sh`,
`tazamun-installer.ps1`, `source.tar.gz`, and provenance attestations; the
formula lands in `cc1a2b/homebrew-tap` and the npm package goes live. Watch
it with:

```bash
gh run watch $(gh run list --workflow=Release --limit 1 --json databaseId --jq '.[0].databaseId')
```

If a job sits **queued for more than a minute or two**, the runner label is
wrong — that is exactly how the old workflow died, silently, for a week.

## 7. Prove the README is telling the truth

Every install path the README advertises, run for real:

```bash
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/cc1a2b/tazamun/releases/latest/download/tazamun-installer.sh | sh
brew install cc1a2b/tap/tazamun
npm install -g tazamun && tazamun --version
cargo install tazamun
gh attestation verify tazamun-x86_64-unknown-linux-gnu.tar.gz --repo cc1a2b/tazamun
```

And the self-updater — the point of all this:

```bash
tazamun update --check     # "Already up to date (0.1.0)." proves discovery + matching
```

The full download-and-swap path gets its first honest exercise when v0.1.1
exists; until then `--check` proves release discovery, asset matching and
version comparison.

## 8. Every release after this one

1. Bump `version` in `Cargo.toml`; `cargo build` once so `Cargo.lock` follows.
2. Rewrite `RELEASE_NOTES.md` for the version.
3. Gates: `cargo fmt --check && cargo clippy --all-targets -- -D warnings && cargo test`.
4. Commit, push, tag `vX.Y.Z`, push the tag. Everything but crates.io is
   automatic. `cargo publish` for the crate.
5. Spot-check `tazamun update` from the previous version — this is the release
   that proves the updater end to end.

Never delete a published release: `tazamun update` and both installers
resolve `releases/latest`, and old binaries update by finding newer tags.

## Troubleshooting

- **Release job queued forever** — dead runner label. `gh run view --log` will
  show it never started; re-check `runs-on` values against currently offered
  hosted runners.
- **npm publish fails with E403** — the token is not an Automation token, or
  the name was claimed between check and publish. `npm view tazamun` to see
  who holds it.
- **Homebrew job fails to push** — the PAT lacks Contents write on the tap
  repo, or the secret name is not exactly `HOMEBREW_TAP_TOKEN`.
- **`tazamun update` says 404** — no releases yet, or the repo went private
  again; with a private repo it needs `--token` / `GITHUB_TOKEN`.
- **A user reports the updater "downloads then errors"** — that is the
  archive-layout contract. `cargo test update_bin_path` must pass against the
  layout of the release that user is updating *to*.
